# VoiceOrder 모델 선택 및 아키텍처 정리

## 1. 문서 목적

이 문서는 현재 VoiceOrder 프로젝트에서 사용 중인 음성 인식 방식, 주문 해석 방식, 저장 구조를 정리하고, 각 선택의 이유와 장단점, 이후 확장 방향을 기록하기 위한 문서다.

이 프로젝트는 현재 "빠르게 동작하는 웹 키오스크 프로토타입"을 목표로 하고 있으므로, 운영 규모의 고정밀 AI 시스템보다는 구현 속도와 데모 적합성을 우선한 구조를 채택하고 있다.

## 2. 현재 아키텍처 한눈에 보기

```text
사용자 음성 입력
  -> 브라우저 Web Speech API
  -> 텍스트 변환 결과(transcript)
  -> Flask 서버로 전송
  -> 규칙 기반 메뉴/수량 파싱
  -> 장바구니 반영
  -> 주문 확정 시 SQLite 저장
```

## 3. 현재 선택된 방식

### 3-1. 음성 인식

- 현재 선택: 브라우저 `Web Speech API`
- 적용 위치: 프론트엔드 (`templates/index.html`)
- 역할: 사용자의 마이크 입력을 텍스트로 변환

#### 이 방식을 선택한 이유

- 별도의 STT 서버를 직접 운영하지 않아도 된다.
- Flask 백엔드에 무거운 ML 추론 파이프라인을 붙이지 않아도 된다.
- 브라우저에서 바로 음성 인식을 시작할 수 있어 데모 구현 속도가 빠르다.
- 키오스크 프로토타입 단계에서 "음성 주문이 된다"는 사용자 경험을 빠르게 검증할 수 있다.

#### 장점

- 구현이 단순하다.
- 별도 GPU나 모델 서빙 인프라가 필요 없다.
- 초기 개발 비용과 운영 비용이 낮다.
- 실시간 음성 입력 UX를 짧은 코드로 붙일 수 있다.

#### 단점

- 실제 음성 인식 모델이 브라우저/플랫폼 내부에 숨겨져 있어 제어가 어렵다.
- 브라우저 종류와 버전에 따라 품질 차이가 날 수 있다.
- 마이크 권한, 브라우저 지원 여부, 네트워크 상태에 영향을 받는다.
- 프로젝트가 직접 STT 모델을 선택하거나 튜닝하는 구조가 아니다.
- 전문 용어, 메뉴 별칭, 매장별 발화 습관을 세밀하게 반영하기 어렵다.

### 3-2. 주문 해석

- 현재 선택: 규칙 기반 문자열 매칭
- 적용 위치: 백엔드 (`app.py`)
- 역할: 텍스트에서 메뉴명과 수량을 추출해 주문 후보 생성

#### 이 방식을 선택한 이유

- 현재 메뉴 수가 적고 구조가 단순하다.
- 데모 단계에서는 예측 불가능한 AI 출력보다 결정적인 규칙이 디버깅에 유리하다.
- 비용 없이 빠르게 동작하는 서버 로직이 필요하다.

#### 장점

- 동작이 예측 가능하다.
- 응답 속도가 빠르다.
- 추가 API 비용이 없다.
- 버그가 생겨도 원인을 코드 수준에서 추적하기 쉽다.

#### 단점

- 정확히 일치하지 않는 표현에 약하다.
- 메뉴 별칭이 늘어날수록 수동 관리 비용이 커진다.
- 복합 문장이나 자연스러운 주문 문장을 깊게 이해하지 못한다.
- 유사 메뉴 추천, 카테고리 추론, 문맥 기반 보정이 어렵다.

### 3-3. 저장 구조

- 현재 선택: SQLite
- 적용 위치: `init_db.py`, `kiosk.db`
- 역할: 메뉴, 주문, 주문 상세 저장

#### 이 방식을 선택한 이유

- 로컬 개발과 데모에 적합하다.
- 별도 DB 서버를 준비하지 않아도 된다.
- 초기 세팅이 단순하다.

#### 장점

- 가볍고 빠르다.
- 프로젝트 실행 장벽이 낮다.
- 초기 시연이나 발표용으로 적합하다.

#### 단점

- 동시성, 운영 확장성, 배포 구조 측면에서 한계가 있다.
- 운영 환경에서는 파일 기반 DB 관리가 불편할 수 있다.

## 4. 현재 구조를 선택한 핵심 판단

현재 VoiceOrder는 "정확도 최고"보다 "짧은 시간 안에 작동하는 음성 주문 웹 키오스크"를 만드는 것이 더 중요하다.

따라서 현재 구조는 다음 우선순위를 따른다.

1. 빠른 구현
2. 낮은 운영 복잡도
3. 데모 가능한 사용자 경험
4. 이후 AI 확장이 가능한 구조 확보

즉, 지금은 브라우저 STT + Flask + 규칙 기반 파서 + SQLite 조합이 가장 현실적인 출발점이다.

## 5. 왜 아직 LLM 기반 주문 해석을 쓰지 않았는가

LLM을 사용하면 다음과 같은 장점이 있다.

- "아메리카노 같은 차가운 커피 하나" 같은 자연어 이해
- 유사 메뉴 추천
- 잘못 인식된 단어 보정
- 카테고리 추론

하지만 현재 단계에서 바로 도입하지 않은 이유는 다음과 같다.

- API 비용이 발생한다.
- 응답 시간이 현재보다 길어질 수 있다.
- 프롬프트 설계와 예외 처리가 추가로 필요하다.
- 결과가 항상 결정적이지 않을 수 있다.
- 메뉴 데이터 구조와 프론트 UI가 먼저 정리되어야 활용 가치가 커진다.

즉, LLM은 유용하지만 현재 프로젝트의 1차 목표인 "프로토타입 완성" 이후에 붙이는 것이 더 합리적이다.

## 6. 향후 확장 아키텍처

정확히 일치하는 메뉴가 없을 때 유사 메뉴 또는 카테고리로 안내하려면 다음 단계 구조가 유력하다.

```text
사용자 음성
  -> Web Speech API
  -> transcript
  -> Flask 서버
  -> 1차 규칙 기반 정확 매칭
  -> 실패 시 유사도 분석 계층
      -> 후보 메뉴 추천
      -> 관련 카테고리 추론
  -> 프론트엔드에 추천 메뉴/카테고리 반환
  -> 해당 섹션 또는 메뉴 카드로 스크롤 이동
```

### 필요한 선행 조건

- 메뉴에 카테고리 정보가 있어야 한다.
- 프론트 UI에 카테고리 섹션 또는 이동 가능한 앵커 구조가 있어야 한다.
- 서버 응답에 "추천 메뉴", "추천 카테고리", "이동 대상 id" 같은 필드가 추가되어야 한다.

## 7. 미래 후보 비교

### 옵션 A. 현재 방식 유지

- 설명: Web Speech API + 규칙 기반 파서 유지
- 적합한 경우: 데모, 발표, 소규모 실습, 빠른 구현
- 한계: 유연한 자연어 해석과 추천 기능이 약함

### 옵션 B. STT는 유지하고 주문 해석만 LLM 보강

- 설명: 음성 텍스트 변환은 그대로 두고, 파싱 실패 시 LLM으로 유사 메뉴 또는 카테고리 추천
- 적합한 경우: 현재 구조를 크게 바꾸지 않고 정확도를 높이고 싶을 때
- 장점: 기존 코드와 결합이 쉽다.
- 단점: 외부 API 비용과 예외 처리 부담이 추가된다.

### 옵션 C. 서버 기반 STT + LLM 해석

- 설명: 브라우저 의존도를 줄이고 서버 또는 외부 API 기반 STT와 LLM을 함께 사용
- 적합한 경우: 운영 서비스 수준의 정확도와 통제가 필요할 때
- 장점: 더 많은 제어권과 확장성을 가질 수 있다.
- 단점: 구현/운영 복잡도와 비용이 크게 증가한다.

## 8. 현재 시점의 권장 결론

현재 프로젝트 기준으로 가장 적절한 판단은 다음과 같다.

- 음성 인식은 당분간 `Web Speech API`를 유지한다.
- 주문 해석은 현재 규칙 기반 방식을 유지한다.
- 정확 매칭 실패 시에만 보조적으로 유사 메뉴 추천 계층을 추가하는 방향이 가장 현실적이다.
- 이 추천 계층은 메뉴 카테고리와 프론트 UI 구조가 정리된 뒤 도입하는 것이 맞다.

즉, 지금의 선택은 "최종 형태"가 아니라 "프로토타입에 최적화된 1단계 아키텍처"라고 보는 것이 정확하다.

## 9. 한줄 요약

VoiceOrder는 현재 브라우저 기반 음성 인식과 규칙 기반 주문 해석을 사용하는 가벼운 프로토타입 구조이며, 향후에는 정확 매칭 실패 시 LLM 또는 유사도 분석 계층을 추가해 추천 및 화면 이동 기능으로 확장하는 것이 적절하다.
